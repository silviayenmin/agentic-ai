# QA Engineer (QA) Agent

## Role
The QA Engineer agent is responsible for thoroughly testing the codebase generated by the Developer agent. It identifies bugs, logic errors, and discrepancies with the initial requirements, providing a structured report.

## Responsibilities
-   Analyze the generated source code for correctness and completeness against project requirements.
-   Identify and list specific bugs or missing features.
-   Determine if the codebase passes or fails its quality assessment.
-   Provide a structured QA report that can be used by the Developer for fixes.

## What it Does
-   Receives the current `codebase` from the Developer agent.
-   Uses an LLM (based on user's `provider` and `model` choice) to evaluate the code.
-   Generates a JSON `qa_report` containing a `status` (PASSED/FAILED) and a `bugs` list.
-   The system uses this report to decide whether to loop back to the Developer or proceed to the Monitor.

## Input Data
-   `state["codebase"]` (dict): The current state of the codebase (filename -> content), as generated by the Dev agent.
-   `state["project_requirements"]` (string): (Context for the LLM) The overall project goal.
-   `state["provider"]` (string): The chosen LLM provider.
-   `state["model"]` (string): The specific LLM model ID.

## Output Data (Transfers to Dev Agent or Monitor Agent)
-   `state["qa_report"]` (dict): A dictionary with `status` (PASSED/FAILED) and a list of `bugs` (strings).
-   `state["logs"]` (List[str]): Appends log entries detailing its activity.
-   `state["agent_statuses"]["QA"]` (string): Updates its status to "working" then "done".
-   `state["current_agent"]` (string): Sets the current active agent to "QA".
-   **Conditional Routing**: Based on `qa_report["status"]`, it directs the flow back to "Dev" (if FAILED) or to "Monitor" (if PASSED).
